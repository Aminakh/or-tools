00014 // The aim is to provide a basis for solving knapsack problems:
00015 //   - 0-1 knapsack problem,
00016 //   - Multi-dimensional knapsack problem,
00017 //   - TODO(user) Multi-dimensional knapsack problem with n-ary conflicts
00018 //     between items.
00019 //
00020 // Given n items, each with a profit and a weight, given a knapsack of
00021 // capacity c, the goal is to find a subset of items which fits inside c
00022 // and maximizes the total profit.
00023 // The knapsack problem can easily be extended from 1 to d dimensions.
00024 // As an example, this can be useful to constrain the maximum number of
00025 // items inside the knapsack.
00026 // Without loss of generality, profits and weights are assumed to be positive.
00027 //
00028 // From a mathematical point of view, the multi-dimensional knapsack problem
00029 // can be modeled by d linear constraints:
00030 // ForEach(j:1..d)(Sum(i:1..n)(weight_ij * item_i) <= c_j
00031 // where item_i is a 0-1 integer variable.
00032 // Then the goal is to maximize: Sum(i:1..n)(profit_i * item_i).
00033 //
00034 // There are several ways to solve knapsack problems. One of the most
00035 // efficient ways is based on dynamic programming (mainly when weights, profits
00036 // and dimensions are small, the algorithm runs in pseudo polynomial time).
00037 // Unfortunately when adding conflict constraints the problem becomes strongly
00038 // NP-hard, i.e. there is no pseudo-polynomial algorithm to solve it.
00039 // That's the reason why the most of the following code is based on branch and
00040 // bound search.
00041 //
00042 // For instance to solve a 2-dimension knapsack problem with 9 items,
00043 // one just has to feed a profit vector with the 9 profits, a vector of 2
00044 // vectors for weights, and a vector of capacities.
00045 // E.g.:
00046 //   vector: profits = [1, 2, 3, 4, 5, 6, 7, 8, 9]
00047 //   vector of vector: weights = [ [1, 2, 3, 4, 5, 6, 7, 8, 9],
00048 //                                 [1, 1, 1, 1, 1, 1, 1, 1, 1]]
00049 //   vector: capacities = [34, 4]
00050 // And then:
00051 //   KnapsackSolver solver(KnapsackSolver::KNAPSACK_MULTIDIMENSION_SOLVER,
00052 //                         "Multi-dimensional solver");
00053 //   solver.Init(profits, weights, capacities);
00054 //   int64 profit = solver.Solve();

00074 // ----- KnapsackSolver -----
00075 // KnapsackSolver is a factory for knapsack solvers. Several solvers are
00076 // implemented, some can deal with a limited number of items, some can deal with
00077 // several dimensions...
00078 // Currently 4 algorithms are implemented:
00079 //  - KNAPSACK_BRUTE_FORCE_SOLVER: Limited to 30 items and one dimension, this
00080 //    solver uses a brute force algorithm, ie. explores all possible states.
00081 //    Experiments show competitive performance for instances with less than
00082 //    15 items.
00083 //  - KNAPSACK_64ITEMS_SOLVER: Limited to 64 items and one dimension, this
00084 //    solver uses a branch & bound algorithm. This solver is about 4 times
00085 //    faster than KNAPSACK_MULTIDIMENSION_SOLVER.
00086 //  - KNAPSACK_DYNAMIC_PROGRAMMING_SOLVER: Limited to one dimension, this solver
00087 //    is based on a dynamic programming algorithm. The time and space
00088 //    complexity is O(capacity * number_of_items).
00089 //  - KNAPSACK_MULTIDIMENSION_BRANCH_AND_BOUND_SOLVER: This solver can deal
00090 //    with both large number of items and several dimensions. This solver is
00091 //    based on branch and bound.
00092 //  - KNAPSACK_MULTIDIMENSION_CBC_MIP_SOLVER: This solver can deal with both
00093 //    large number of items and several dimensions. This solver is based on
00094 //    Integer Programming solver CBC.
00095 //  - KNAPSACK_MULTIDIMENSION_GLPK_MIP_SOLVER: This solver can deal with both
00096 //    large number of items and several dimensions. This solver is based on
00097 //    Integer Programming solver GLPK.
00098 //
00099 // KnapsackSolver also implements a problem reduction algorithm based on lower
00100 // and upper bounds (see Ingargolia and Korsh - A reduction algorithm for
00101 // zero-one single knapsack problems. Management Science 1973). This reduction
00102 // method is preferred to better algorithms (see for instance Martello and Toth
00103 // - A new algorithm for the 0-1 knapsack problem. Management Science 1988),
00104 // because it remains valid with more complex problems, eg. multi-dimensional,
00105 // conflicts...
00106 // The main idea is to compute lower and upper bounds for each item in or out
00107 // of the knapsack; if the best lower bound is strictly greater than the upper
00108 // bound when an item is in, then this item is surely not in the optimal
00109 // solution.

00161 // The following code defines needed classes for the KnapsackGenericSolver
00162 // class which is the entry point to extend knapsack with new constraints such
00163 // as conflicts between items.
00164 //
00165 // Constraints are enforced using KnapsackPropagator objects, in the current
00166 // code there is one propagator per dimension (KnapsackCapacityPropagator).
00167 // One of those propagators, named master propagator, is used to guide the
00168 // search, i.e. decides which item should be assigned next.
00169 // Roughly speaking the search algorithm is:
00170 //  - While not optimal
00171 //    - Select next search node to expand
00172 //    - Select next item_i to assign (using master propagator)
00173 //    - Generate a new search node where item_i is in the knapsack
00174 //      - Check validity of this new partial solution (using propagators)
00175 //      - If valid, add this new search node to the search
00176 //    - Generate a new search node where item_i is not in the knapsack
00177 //      - Check validity of this new partial solution (using propagators)
00178 //      - If valid, add this new search node to the search
00179 //
00180 // TODO(user): Add a new propagator class for conflict constraint.
00181 // TODO(user): Add a new propagator class used as a guide when the problem has
00182 // several dimensions.

00196 // ----- KnapsackItem -----
00197 // KnapsackItem is a small struct to pair an item weight with its
00198 // corresponding profit.
00199 // The aim of the knapsack problem is to pack as many valuable items as
00200 // possible. A straight forward heuristic is to take those with the greatest
00201 // profit-per-unit-weight. This ratio is called efficiency in this
00202 // implementation. So items will be grouped in vectors, and sorted by
00203 // decreasing efficiency.
00204 // Note that profits are duplicated for each dimension. This is done to
00205 // simplify the code, especially the GetEfficiency method and vector sorting.
00206 // As there usually are only few dimensions, the overhead should not be an
00207 // issue.

00209   KnapsackItem(int _id, int64 _weight, int64 _profit)
00210       : id(_id),
00211         weight(_weight),
00212         profit(_profit) {
00213   }
00214   double GetEfficiency(int64 profit_max) const {
00215     return (weight > 0) ?
00216         static_cast<double>(profit) / static_cast<double>(weight) :
00217         static_cast<double>(profit_max);
00218   }

00221   // communicate with other propagators and state.

00274 // ----- KnapsackSearchPath -----
00275 // KnapsackSearchPath is a small class used to represent the path between a
00276 // node to another node in the search tree.
00277 // As the solution state is not stored for each search node, the state should
00278 // be rebuilt at each node. One simple solution is to apply all decisions
00279 // between the node 'to' and the root. This can be computed in
00280 // O(number_of_items).
00281 // However it is possible to achieve better average complexity. Two
00282 // consecutively explored nodes are usually close enough (ie. much less than
00283 // number_of_items) to benefit from an incremental update from the node
00284 // 'from' to the node 'to'.

00286 // So the state can be built by reverting all decisions from 'from' to 'via'
00287 // and applying all decisions from 'via' to 'to'.

00291                      const KnapsackSearchNode& to);

00294   const KnapsackSearchNode& via() const { return *via_; }

00296   const KnapsackSearchNode* MoveUpToDepth(const KnapsackSearchNode& node,
00297                                           int depth) const;
00298  private:
00299   const KnapsackSearchNode& from_;

00407 // ----- KnapsackCapacityPropagator -----
00408 // KnapsackCapacityPropagator is a KnapsackPropagator used to enforce
00409 // a capacity constraint.
00410 // As a KnapsackPropagator is supposed to compute profit lower and upper
00411 // bounds, and get the next item to select, it can be seen as a 0-1 Knapsack
00412 // solver. The most efficient way to compute upper bound is to iterate on
00413 // items in profit-per-unit-weight decreasing order. The break item is
00414 // commonly defined as the first item for which there is not enough remaining
00415 // capacity. Selecting this break item as the next-item-to-assign usually
00416 // gives the best results (see Greenbreg & Hegerich).
00417 // This is exactly what is implemented in this class.
00418 // When there is only one propagator, it is possible to compute a better
00419 // profit lower bound almost for free. During the scan to find the
00420 // break element all unbound items are added just as if they were part of
00421 // the current solution. This is used both in ComputeProfitBounds and
00422 // CopyCurrentSolutionPropagator.
00423 // For incrementality reasons, the ith item should be accessible in O(1). That's
00424 // the reason why item vector has to be duplicated 'sorted_items_'.
00425 class KnapsackCapacityPropagator : public KnapsackPropagator {
00426  public:
00427   KnapsackCapacityPropagator(const KnapsackState& state, int64 capacity);
00428   virtual ~KnapsackCapacityPropagator();
00429   virtual void ComputeProfitBounds();
00430   virtual int GetNextItemId() const { return break_item_id_; }
00431 
00432  protected:
00433   // Initializes KnapsackCapacityPropagator (eg. sort items in decreasing
00434   // order).
00435   virtual void InitPropagator();
00436   // Updates internal data structure incrementally (ie. 'consumed_capacity_')
00437   // to avoid a O(number_of_items) scan.
00438   virtual bool UpdatePropagator(bool revert,
00439                                 const KnapsackAssignment& assignment);
00440   virtual void CopyCurrentStateToSolutionPropagator(
00441       std::vector<bool>* solution) const;
00442 
00443  private:
00444   // An obvious additional profit upper bound corresponds to the linear
00445   // relaxation: remaining_capacity * efficiency of the break item.
00446   // It is possible to do better in O(1), using Martello-Toth bound U2.
00447   // The main idea is to enforce integrality constraint on the break item,
00448   // ie. either the break item is part of the solution, either it is not.
00449   // So basically the linear relaxation is done on the item before the break
00450   // item, or the one after the break item.

00452   int64 GetAdditionalProfit(int64 remaining_capacity, int break_item_id) const;
