00014 // An implementation of a cost-scaling push-relabel algorithm for
00015 // the min-cost flow problem.
00016 //
00017 // In the following, we consider a graph G = (V,E) where V denotes the set
00018 // of nodes (vertices) in the graph, E denotes the set of arcs (edges).
00019 // n = |V| denotes the number of nodes in the graph, and m = |E| denotes the
00020 // number of arcs in the graph.
00021 //
00022 // To each arc (v,w) is associated a capacity c(v,w) and a unit cost cost(v,w).
00023 // To each node v is associated quantity named supply(v), which represents a
00024 // supply of fluid (if >0) or a demand (if <0).
00025 // Furthermore, no fluid are created in the graph so
00026 //    sum on v in V supply(v) = 0
00027 //
00028 // A flow is a function from E to R such that:
00029 // a) f(v,w) <= c(v,w) for all (v,w) in E (capacity constraint).
00030 // b) f(v,w) = -f(w,v) for all (v,w) in E (flow antisymmetry constraint).
00031 // c) sum on v f(v,w) = 0  (flow conservation.
00032 //
00033 // The cost of a flow is sum on (v,w) in E ( f(v,w) * cost(v,w) )
00034 //
00035 // The problem to solve is to find a flow of minimum cost such that all the
00036 // fluid flows from the supply nodes to the demand nodes.
00037 //
00038 // The principles behind this algorithm are the following:
00039 //  1/ handle pseudo-flows instead of flows and refine pseudo-flows until an
00040 // epsilon-optimal minimum-cost flow is obtained,
00041 //  2/ deal with epsilon-optimal pseudo-flows.
00042 //
00043 // 1/ A pseudo-flow is like a flow, except that the inflow can be different from
00044 // the outflow. If it is the case at a given node v, it is said that there is an
00045 // excess (or deficit) at node v. A deficit is denoted by a negative excess
00046 // and inflow = outflow + excess.
00047 // (Look at graph/max_flow.h to see that the definition
00048 // of preflow is more restrictive than the one for pseudo-flow in that a preflow
00049 // only allows non-negative excesses, i.e. no deficit.)
00050 // More formally, a pseudo-flow is a function f such that:
00051 // a) f(v,w) <= c(v,w) for all (v,w) in E  (capacity constraint).
00052 // b) f(v,w) = -f(w,v) for all (v,w) in E (flow antisymmetry constraint).
00053 //
00054 // For each v in E, we also define the excess at node v, the algebraic sum of
00055 // all the incoming preflows at this node, added together with the supply at v.
00056 //    excess(v) = sum on u f(u,v) + supply(v)
00057 //
00058 // The goal of the algorithm is to obtain excess(v) = 0 for all v in V, while
00059 // consuming capacity on some arcs, at the lowest possible cost.
00060 //
00061 // 2/ Each node has an associated "price" (or potential), in addition to its
00062 // excess. It is formally a function from E to R (the set of real numbers.).
00063 // For a given price function p, the reduced cost of an arc (v,w) is:
00064 //    cost_p(v,w) = cost(v,w) + p(v) - p(w)
00065 // (cost(v,w) is the cost of arc (v,w) .)
00066 //
00067 // For a constant epsilon >= 0, a pseudo-flow f is said to be epsilon-optimal
00068 // with respect to a pricing function p if for all residual arc (v,w) in E,
00069 //    cost_p(v,w) >= -epsilon.
00070 //
00071 // A flow f is optimal if and only if there exists a price function p such that
00072 // no arc is admissible with respect to f and p.
00073 //
00074 // If the arc costs are integers, and epsilon < 1/n, any epsilon-optimal flow
00075 // is optimal. The integer cost case is handled by multiplying all the arc costs
00076 // and the initial value of epsilon by (n+1). When epsilon reaches 1, and
00077 // the solution is epsilon-optimal, it means: for all residual arc (v,w) in E,
00078 //    (n+1) * cost_p(v,w) >= -1, thus cost_p(v,w) >= -1/(n+1) >= 1/n, and the
00079 // solution is optimal.
00080 //
00081 // A node v is said to be *active* if excess(v) > 0.
00082 // In this case the following operations can be applied to it:
00083 // - if there are *admissible* incident arcs, i.e. arcs which are not saturated,
00084 //   and whose reduced costs are negative, a PushFlow operation can
00085 //   be applied. It consists in sending as  much flow as both the excess at the
00086 //   node and the capacity of the arc permit.
00087 // - if there are no admissible arcs, the active node considered is relabeled,
00088 // This is implemented in Discharge, which itself calls PushFlow and Relabel.
00089 //
00090 // Discharge itself is called by Refine. Refine first saturates all the
00091 // admissible arcs, then builds a stack of active nodes. It then applies
00092 // Discharge for each active node, possibly adding new ones in the process,
00093 // until no nodes are active. In that case an epsilon-optimal flow is obtained.
00094 //
00095 // Optimize iteratively calls Refine, while epsilon > 1, and divides epsilon by
00096 // alpha (set by default to 5) before each iteration.
00097 //
00098 // The algorithm starts with epsilon = C, where C is the maximum absolute value
00099 // of the arc costs. In the integer case which we are dealing with, since all
00100 // costs are multiplied by (n+1), the initial value of epsilon is (n+1)*C.
00101 // The algorithm terminates when epsilon = 1, and Refine() has been called.
00102 // In this case, a minimum-cost flow is obtained.
00103 //
00104 // The complexity of the algorithm is O(n^2*m*log(n*C)) where C is the value of
00105 // the largest arc cost in the graph.
00106 //
00107 // The starting reference for this class of algorithms is:
00108 // A.V. Goldberg and R.E. Tarjan, "Solving Minimum-Cost Flow Problems by
00109 // Successive Approximation." Proceedings of ACM STOC'87, 1987:7-18.
00110 // http://portal.acm.org/citation.cfm?id=28397&CFID=10541307&CFTOKEN=40586317
00111 //
00112 // Implementation issues are tackled in:
00113 // A.V. Goldberg, "An Efficient Implementation of a Scaling Minimum-Cost Flow
00114 // Algorithm," Journal of Algorithms, (1997) 22:1-29
00115 // http://citeseerx.ist.psu.edu/viewdoc/summary?doi=10.1.1.31.258
00116 //
00117 // A.V. Goldberg and M. Kharitonov, "On Implementing Scaling Push-Relabel
00118 // Algorithms for the Minimum-Cost Flow Problem", Network flows and matching:
00119 // First DIMACS implementation challenge, DIMACS Series in Discrete Mathematics
00120 // and Theoretical Computer Science, (1993) 12:157-198.
00121 // ftp://dimacs.rutgers.edu/pub/netflow/...mincost/scalmin.ps
00122 // and in:
00123 // ﻿U. Bunnagel, B. Korte, and J. Vygen. “Efficient implementation of the
00124 // Goldberg-Tarjan minimum-cost flow algorithm.” Optimization Methods and
00125 // Software (1998) vol. 10, no. 2:157-174.
00126 // http://citeseerx.ist.psu.edu/viewdoc/summary?doi=10.1.1.84.9897
00127 //
00128 // We have tried as much as possible in this implementation to keep the
00129 // notations and namings of the papers cited above, except for 'demand' or
00130 // 'balance' which have been replaced by 'supply', with the according sign
00131 // changes to better accomodate with the API of the rest of our tools. A demand
00132 // is denoted by a negative supply.
00133 //

00135 // problems.
